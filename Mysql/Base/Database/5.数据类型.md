## 数据类型
### 1.数值型
```
# 整形
TINYINT             # -128 ~ 127
SMALLINT            # -32768 ~ 32767
MEDIUMINT           # -8388608 ~ 8388607
INT                 # -2,147,483,648 ~ 2,147,483,647
BIGINT

# 浮点型
FLOAT
DOUBLE(长度，小数位数)
DECIMAL

# 定点数
# decimal(p,d) 存储小数的精确值
amount DECIMAL(6,2);
amount列最多可以存储6位数字，小数位数为2位; 
因此，amount列的范围是从-9999.99到9999.99

# unsigned  限定正数
只用作整型和浮点型中，使得数据不许出现负数
int -> -128 ~ 127
int unsigned -> 0 ~ 255，并且能够让正数的存储上限翻倍

# 什么时候用 UNSIGNED 最合适？
自增主键/ID（从 1 开始递增，不需要负数）：INT UNSIGNED AUTO_INCREMENT
计数类字段：like 数、访问量、库存数量、人数、次数、页码等（天然不该出现负数）
状态码/枚举编号：如 status、type、level
时间/长度/大小（如果业务定义上不可能为负）：时长秒数、文件大小等
```

### 2.字符型
```
# 字符串
MySQL 里“长度”写的是字符数，但真正占空间的是字节数，而且跟字符集有关。
CHAR(n)  			# 储存恒定(n),不够补空格,n指字符数，0~255字节
VARCHAR(n)  		# 储存实际长度n+1~2,存储的是字符数

# 文本类型
TINYTEXT            # 0~255
TEXT                # 65,535 字节（≈64KB）
MEDIUMTEXT          # 16,777,215 字节（≈16MB）
LONGTEXT            # 4,294,967,295 字节（≈4GB）

# 二进制类型
一般用来存储图片、音乐、视频等
TINYBLOB
BLOB
MEDIUMBLOB

# 数据库规范通常不推荐使用 BLOB 和 TEXT 类型，这两种类型具有一些缺点和限制：
不能有默认值。
在使用临时表时无法使用内存临时表，只能在磁盘上创建临时表（《高性能 MySQL》书中有提到）。
检索效率较低。不能直接创建索引，需要指定前缀长度。
可能会消耗大量的网络和 IO 带宽。
可能导致表上的 DML 操作变慢。
```

### 3. 时间类型
```
# 日期时间
YEAR                        # 年
DATE                        # 年月日
TIME                        # 时分秒
DATETIME                    # 年月日时分秒
TIMESTAMP                   # 有时区信息
DATETIME：'1000-01-01 00:00:00.000000' 到 '9999-12-31 23:59:59.999999'
Timestamp：'1970-01-01 00:00:01.000000' UTC 到 '2038-01-19 03:14:07.999999' UTC
```

# 特殊值
```
null 空值
1.null没有意义，不属于任何一种数据类型
2.null出现在表中，mysql会对它进行特殊处理，使其拥有所在字段的数值类型，以便增删改查能够顺利进行
* NULL 的存储空间占用取决于数据库的实现，通常需要一些空间来标记该值为空。
* '' 的存储空间占用通常较小，因为它只存储一个空字符串的标志，不需要存储实际的字符。

# 布尔类型
select 1 = true;
select 0 = false;
```
